# Инструкция: планирование и старт задач в Jira (Partenit Claudev)

Краткие правила для людей и для настройки досок. Детали сервиса — в `.env` и `README.md`.

---

## 1. Какие типы задач принимает вебхук

Родительская задача должна быть одним из типов:

- **Task**, **Bug**, **Story**
- или русские эквиваленты: **Задача**, **Баг**, **История**

Подзадачи (**Sub-task**) используются для стадий пайплайна — их создаёт сам сервис.

---

## 2. Триггер запуска пайплайна

- Перевести задачу в статус **`In Progress`** (или значение переменной `TRIGGER_STATUS` в `.env`).
- Должен быть настроен webhook Jira на URL сервиса:  
  `POST /webhook/jira?secret=<WEBHOOK_SECRET>`.

---

## 3. Режимы по заголовку (Summary)

| Префикс в Summary | Назначение |
|-------------------|------------|
| *(нет префикса)* | Обычный dev-пайплайн: анализ → архитектура → разработка → тесты |
| **`PLAN:`** | Планирование в Jira: LLM разбивает фичу на эпики и задачи в проекте |
| **`BOOTSTRAP:`** | Greenfield: продуктовый бриф, baseline архитектуры и governance, скелет репо, разбиение на эпики/стори/таски; затем стандартный dev-пайплайн |

Префиксы чувствительны к регистру в коде — используйте **`PLAN:`** и **`BOOTSTRAP:`** как в примерах.

**Важно:** статус колонки в Jira **`План` / `Plan` и т.п. не запускает пайплайн** — вебхук принимает только перечисленные в §6 статусы (в первую очередь **`In Progress`**). Префикс **`PLAN:`** задаётся в **Summary**, а триггер — всё равно перевод в **`In Progress`**.

---

## 4. Минимальный набор полей в тикете

**Достаточно для технического старта:**

1. **Тип** — Task / Bug / Story (или русские названия выше).
2. **Summary** — текст; для нужного режима добавить **`PLAN:`** или **`BOOTSTRAP:`** в начало.
3. **Статус** — перевести в **`In Progress`** (или `TRIGGER_STATUS`).

**Желательно:**

- **Description** — для `BOOTSTRAP`, `PLAN` и любых задач без описания модель будет хуже ориентироваться. Имеет смысл указать: цель, пользователей, MVP, ограничения (стек, хостинг, безопасность).

Необязательно для старта: labels, компоненты, приоритет (сервис может досыпать labels сам).

---

## 5. Первые тикеты для инициации проекта

### Greenfield (новый проект)

1. Одна **родительская** задача: **Summary** = `BOOTSTRAP: <название продукта>`.
2. Заполнить **Description** по возможности.
3. Перевести в **`In Progress`**.
4. В GitHub должен существовать целевой репозиторий из `GITHUB_REPO` (хотя бы минимальный контент, например README).

### Уже существующий код

1. Обычная задача **без** `BOOTSTRAP:` / `PLAN:`.
2. **`In Progress`** — создаются стандартные подзадачи пайплайна.

### Только разнесение по Jira (без bootstrap-репо)

1. **Summary** = `PLAN: <описание фичи>`.
2. **`In Progress`** — создаются эпики и задачи согласно логике планирования.

---

## 6. Статусы в Jira и `.env`

Имена статусов в Jira должны совпадать с переменными `STATUS_*` и `TRIGGER_STATUS` в `.env` (пробелы и регистр важны).

Типичный набор:

- **To Do**, **In Progress**, **Done**, **Cancelled**
- **In Review**, **Ready to Merge** — по необходимости для ревью и авто-merge PR
- После bootstrap родитель может переводиться в статус из **`AUTO_TRANSITION_ON_BOOTSTRAP_COMPLETE`** (по умолчанию задуман как **Ready for Development** — если такого статуса нет в workflow, задайте другой существующий статус или отключите переменную, оставив пустым)

Команда Telegram **`/cancel PROJ-123`** переводит задачу в **`STATUS_CANCELLED`**; чтобы воркеры остановились, Jira должна доставить webhook на сервис (для локального `localhost` нужен публичный URL / туннель).

---

## 7. Один инстанс — один основной Jira-проект

`JIRA_PROJECT_KEY` в `.env` задаёт проект по умолчанию для JQL при старте, создания части сущностей и подзадач. Токен Jira даёт доступ ко всему сайту, но логика приложения ориентирована на один ключ проекта на деплой.

---

## 8. GitHub

- Репозиторий задаётся как **`owner/repo`** в `GITHUB_REPO`.
- Нужен **токен** (`GITHUB_TOKEN` / `GITHUB_TOKEN_TARGET`) с правами на clone, push, PR.
- Второй репозиторий опционально: лейбл задачи **`repo:bridge`** и переменные `GITHUB_REPO_BRIDGE`, `GITHUB_TOKEN_BRIDGE`.
